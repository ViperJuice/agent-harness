# Detailed plan: goal-ID Increment 2 — planner skills emit/reference EC-IDs (#211)

## Task

Make the planner **skills** emit and reference `EC-<ALIAS>-<N>` goal IDs by default,
so the decidable goal-coverage mechanism shipped in Increment 1 (#247) actually
applies to real roadmaps/plans (not just fixtures). Scope: the skill prose + the
plan-doc validator + the bundle regen. **Out of scope:** bulk-migrating the 9
existing `specs/phase-plans-v*.md` roadmaps — opt-in per roadmap (they stay
`not_applicable` / legacy until re-authored), which keeps this non-breaking.

## Research summary

- Increment 1 (merged) already ships: `roadmap_lint` EC-ID parse + `(H)` reconciliation
  (all-or-none, alias-scoped, unique, gaps-allowed), `goal_coverage.check_goal_coverage`
  (+ `phase_declares_goal_ids`), the `goal-coverage-audit` CLI, and the preflight +
  canonical-closeout gates (warn-default). This increment only makes the SKILLS produce
  conformant roadmaps/plans.
- Skill sources are per-harness trees `skills-src/{claude,codex,gemini,opencode}/<harness>-<skill>/`,
  neutralized + regenerated by `build_bundle` into `phase-loop-skills/` and
  `phase-loop-runtime/src/phase_loop_runtime/skills_bundle/<harness>-<skill>/`. The CANON
  parity test (`test_skills_canon_parity.py`, IF-0-CANON-2 HARD) asserts the committed
  bundle is byte-identical to `build_bundle(skills-src)`. Regen: `phase-loop build-bundle
  --apply`. Edits go in `skills-src/` (all 4 harness copies — same semantic change);
  never hand-edit the generated bundle.
- Exit-criteria emission lives in each harness's `<harness>-phase-roadmap-builder/SKILL.md`
  (+ claude's `references/roadmap-template.md`). Acceptance emission lives in each
  `<harness>-plan-phase/SKILL.md` and `<harness>-plan-detailed/SKILL.md`. The plan-doc
  validator is `<harness>-plan-phase/scripts/validate_plan_doc.py` (claude canonical;
  mirrored into the bundle).

## Changes

### roadmap-builder skill (all 4 harnesses' SKILL.md + claude `references/roadmap-template.md`)

- Change the `**Exit criteria**` emission so **each item leads with a goal ID**:
  `- [ ] EC-<ALIAS>-1 — <Testable assertion. Checkable by shell command or test.>`.
- Add a short "Goal IDs" note: IDs are `EC-<ALIAS>-<N>` (alias = the phase alias);
  **all-or-none per phase** (every exit-criterion carries one, or none do — a mixed
  phase is a `roadmap_lint (H)` error); **unique, alias-scoped, gaps allowed** —
  **never reuse or renumber** a deleted ID (a downstream plan references it by ID; the
  lint allows gaps precisely so a deletion never forces a rebinding renumber). The IDs
  are what let a plan reference a goal instead of restating (and drifting from) it.

### plan-phase + plan-detailed skills (all 4 harnesses' SKILL.md)

- Change acceptance-criteria emission from **restating** the exit-criterion to
  **referencing its EC-ID + naming the proving command**:
  `- [ ] EC-<ALIAS>-<N> — proven by \`<command / test>\``. Replace the "copied or
  refined from the roadmap exit criteria" wording with "reference the roadmap
  exit-criterion's `EC-<ALIAS>-<N>` ID and name the command that proves it; do not
  restate the goal text (the roadmap goal is the single source of truth)."
- Keep the "one contract" rigor rule (acceptance ↔ proving command). Keep
  **plan-internal** acceptance items (no EC-ref) allowed for plan-local done conditions.
- Note the fallback: if the anchored roadmap phase declares **no** EC-IDs (legacy),
  the planner authors acceptance as today (testable assertions) — opt-in per roadmap.

### `validate_plan_doc.py` (claude canonical; regenerated to all harnesses)

- Add a **warn** check (rigor-v1 family, opt-in-to-block like the others): when the
  anchored roadmap phase declares EC-IDs, every declared EC-ID should be referenced by
  an acceptance item, and a referenced EC-ID should exist (no dangling). Reuse
  `goal_coverage` if importable in the validator's runtime; else a lightweight local
  parse mirroring `extract_plan_goal_refs` + `phase_declares_goal_ids`. Legacy phases
  (no EC-IDs) → no finding.

### Regen the bundle

- `phase-loop build-bundle --apply` (or `build_bundle(DEFAULT_SOURCES, ..., apply=True)`),
  committing the regenerated `phase-loop-skills/` + `skills_bundle/` alongside the
  `skills-src/` edits so the CANON parity test stays green.
